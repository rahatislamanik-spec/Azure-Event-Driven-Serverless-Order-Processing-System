🛠️ Issues Faced & Problems Solved

This section documents the major engineering problems encountered during the build, how each issue was diagnosed, the confirmed root cause, and the fix or current status. It is intended to preserve the real troubleshooting history behind the project rather than only showing the final working architecture.

1. Azure for Students Region Restrictions — Phase 1
  • Problem: The original architecture documented Canada Central as the deployment region. Storage Account creation failed with a RequestDisallowedByAzure policy error.
  • Diagnosis Process: Canada East was tested as a fallback and produced the same policy error. The failure was confirmed through both Azure Portal and Azure CLI, proving it was a subscription-level policy restriction rather than a single failed deployment attempt.
  • Root Cause: Azure for Students subscriptions use a curated list of allowed regions. Canada Central and Canada East were not available for the required resource deployment.
  • Fix: Professor Ali Ziyaei was contacted with the issue and approved East US as the permanent project region. All subsequent project resources were deployed to East US.
2. Resource Provider Registration Delays — Phase 1 and recurring
  • Problem: Multiple resource types, including Storage, Network, Web, and Communication resources, failed on first deployment attempt because the related Azure resource provider was not registered.
  • Diagnosis Process: The Azure CLI suggested auto-registration, but immediate retries continued to fail. Rechecking the provider state showed registration was asynchronous and required time to complete.
  • Root Cause: New subscriptions do not always have every resource provider pre-registered. Provider registration can take several minutes even after the registration command is accepted.
  • Fix: A repeatable provider registration pattern was used: run az provider register, then poll registrationState until the provider returns Registered before retrying deployment.
3. Function App GUI Hosting Plan Trap — Phase 1
  • Problem: Python did not appear in the Azure Portal Runtime Stack dropdown while creating the Function App.
  • Diagnosis Process: The issue was traced back to the hosting plan and operating system selection. Selecting Consumption (Windows) silently locked the Function App to Windows.
  • Root Cause: Python Azure Functions require Linux hosting. The portal flow did not make the operating system dependency clear before the selection constrained the available runtime stack.
  • Fix: The portal creation flow was bypassed for this step. Azure CLI was used with --os-type Linux explicitly specified so the Function App could be created with the correct Python runtime.
4. Key Vault RBAC Permissions Gap — Phase 6
  • Problem: Creating secrets in a newly created Key Vault failed with a ForbiddenByRbac error.
  • Diagnosis Process: The Key Vault configuration was inspected and confirmed to use RBAC authorization mode with enableRbacAuthorization: true instead of legacy access policies.
  • Root Cause: In RBAC mode, creating a vault does not automatically grant the creator secret-management permissions. Vault creation permissions and secret data-plane permissions are controlled separately.
  • Fix: The Key Vault Secrets Officer role was explicitly assigned at the vault scope before secret creation was retried.
5. Deployment Overwrite Bug — Phase 4 and Phase 5
  • Problem: Deploying validate_order from a separate folder wiped out the already-working submit_order function. This was confirmed when az functionapp function list showed only validate_order afterward.
  • Diagnosis Process: The function list was checked before and after separate deployments to determine whether the behavior was caching or a real replacement. The result confirmed the deployment package was replacing the whole Function App contents.
  • Root Cause: func azure functionapp publish deploys the current working directory as the complete package for the Function App. Separate function folders caused each deployment to replace what came before it.
  • Fix: All functions were consolidated into one Python V2 function_app.py deployment package using multiple decorators, allowing the full Function App to deploy as one coherent unit.
6. Python Version Mismatch — Phase 3 through Phase 6
  • Problem: Local development defaulted to macOS system Python 3.9.6 while the deployed Azure Function App required Python 3.11.
  • Diagnosis Process: The local python3 --version output was compared against the Function App runtime configuration returned by az functionapp config show.
  • Root Cause: macOS ships an older system Python by default. Nothing automatically aligns the local Python runtime with the Azure Functions runtime.
  • Fix: Python 3.11 was installed through Homebrew and a dedicated .venv311 virtual environment was created for local development and deployment.
7. Missing Dependencies Causing Silent Total Failure — Phase 3 through Phase 6
  • Problem: After deployment, az functionapp function list returned an empty array. No functions were registered, even though the deployment output reported success.
  • Diagnosis Process: Standard deployment output did not show the failure. The issue was traced by comparing requirements.txt against the actual import statements in function_app.py.
  • Root Cause: Required dependencies such as azure-data-tables and azure-communication-email were missing after new functions were added. The remote build installed only listed packages, then the Python worker failed at import time and did not register the functions.
  • Fix: The missing packages were added to requirements.txt. The deployment checklist now includes comparing all imports against requirements.txt before publishing.
8. Queue Message Encoding Bug — Phase 4 Most significant debugging effort
  • Problem: validate_order was correctly coded, correctly registered, and correctly bound to the orders-incoming queue, but every message failed and moved to orders-incoming-poison after reaching MaxDequeueCount. There was no Python traceback or function-level error output.
  • Diagnosis Process: The binding was verified through the local Functions admin API. The storage connection string and target queue were checked. A second consumer process was ruled out. The function was reduced to two log statements and no business logic, but the same failure continued. Azure Functions Core Tools was reinstalled, and the issue still reproduced. The function was then deployed to Azure to rule out a local-only problem. The decisive clue came from Azure Portal Log Stream, which showed: Message decoding has failed! Check MessageEncoding settings.
  • Root Cause: Azure Functions queue triggers default to expecting Base64-encoded queue messages. The Storage Queue Python SDK was sending plain JSON text through queue_client.send_message(json.dumps(...)). The binding layer failed before Python code executed, which explained why no Python-level exception or log line appeared.
  • Fix: Queue message encoding was explicitly configured in host.json with "extensions": { "queues": { "messageEncoding": "none" } }. After redeployment, the live pipeline processed queue messages correctly.
9. submit_order Default Authorization Level — Phase 3
  • Problem: The live submit_order endpoint returned HTTP 401 Unauthorized during the first live test, even though the function worked locally.
  • Diagnosis Process: Local and live behavior were compared. Local testing masked the issue because the local Functions host does not expose the same live authorization behavior by default.
  • Root Cause: The Azure Functions Python V2 route used FUNCTION-level authorization by default, requiring a function key. That was not appropriate for the public customer-facing order submission endpoint.
  • Fix: The route decorator was updated to explicitly set auth_level=func.AuthLevel.ANONYMOUS.
10. Azure Static Web Apps Supported Regions Blocked — RESOLVED: GitHub Pages approved
  • Problem: Static Web App creation failed with RequestDisallowedByAzure in every supported Azure Static Web Apps region tested: Central US, East US 2, West US 2, West Europe, and East Asia.
  • Diagnosis Process: Each supported region was tested individually through Azure CLI. Every attempt returned the same policy error.
  • Root Cause: Azure for Students appears to block Static Web Apps for this subscription, leaving no available supported region for this specific resource type.
  • Resolution: Professor Ali Ziyaei approved GitHub Pages as the substitute. He replied: "Good thinking and good workaround. Yes, it is accepted." (June 22, 2026). Frontend is now live at: https://rahatislamanik-spec.github.io/Azure-Event-Driven-Serverless-Order-Processing-System/
11. Azure Monitor Test Notifications Blocked on Free Tier — Phase 7
  • Problem: Action Group email notification delivery could not be live-tested because intentionally producing a genuine HTTP 5xx error was difficult. The application correctly handled malformed input and returned clean 400 responses instead of crashing.
  • Diagnosis Process: Azure Monitor's built-in az monitor action-group test-notifications create command was attempted to trigger a test notification without forcing a real application failure.
  • Root Cause: The test-notification feature returned (Conflict) Free subscription not supported, indicating a platform limitation for the free/student subscription tier rather than a configuration issue.
  • Status: The alert rule and Action Group were verified by configuration inspection through az monitor metrics alert show and az monitor action-group show. Notification live-fire testing remains blocked by subscription-tier limitations and is documented honestly as configuration-verified rather than live-notification-verified.
12. Quantity Zero Misclassified as Missing Field — Phase 8
  • Problem: During systematic test-matrix validation, an order with quantity: 0 returned the generic missing-field response instead of the clearer "Quantity must be a positive whole number" validation response.
  • Diagnosis Process: The failing case was compared against other invalid-input tests. Missing fields, bad email, insufficient stock, and unknown laptop models all routed correctly, narrowing the issue to the HTTP trigger's required-field check.
  • Root Cause: The required-field check treated all falsy values as missing. In Python, numeric zero is falsy, so 0 was caught before the dedicated quantity validation branch.
  • Fix: The required-field logic was updated to allow 0 through the missing-field check so the quantity-specific validation can return the correct error message. The full six-case test matrix was re-run successfully after the fix.
